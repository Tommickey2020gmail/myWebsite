# 从 Claude Code 到小暖:两类 AI 的记忆哲学之辨

> 本文首发于个人站 <https://tommickey.cn/essays/claude-code-vs-xiaonuan-memory/> ,欢迎转载,请保留原文链接。
>
> 关于"为什么 Claude Code 不用 RAG",以及陪伴型 AI 该怎么"记住"一个人。

## 引子:一个被忽略的反常识

2026 年 1 月,Anthropic 的 Claude Code 团队负责人 Boris Cherny 在 Hacker News 上写下一段话,被转发了上百万次:

> "Claude Code 早期版本用过 RAG + 本地向量数据库,但我们很快发现 agentic search 在大多数情况下都更好。"

在一个"言必称 RAG"的时代,这句话像是从行业主流路线上划下了一道斜线。过去三年里,几乎所有"让大模型记住你的代码库"的方案,都建立在同一套技术栈之上:分块、向量化、向量数据库、语义检索。这是教科书式的标准答案。

而 Claude Code——被认为是 2026 年最强的 AI 编程助手——选择了一条更"原始"的路:**它不建索引,不做向量化,靠 `grep`、`glob`、`find`、`Read` 这些 Unix 老兵般的命令行工具,像一个真正的程序员那样,一行一行地探索代码库**。

一位 Anthropic 工程师在那条帖子下补充道:"In our testing we found that agentic search outperformed [RAG] by a lot, and this was surprising."

这件事值得深思的原因,不在于"RAG 是否过时",而在于:它逼着我们重新审视 **"AI 记忆"这件事究竟该怎么做**。Claude Code 处理的是代码库——高度结构化、有精确语义、有明确边界。而我们正在构建的小暖,是一个面对老人的语音陪伴系统——它要记住的不是函数签名,而是张阿姨上周提过的孙子叫什么、李大爷昨天血压高了三天没说话、王奶奶最近开始反复讲同一个故事。

**两类 AI,两种记忆哲学。Claude Code 的方案能否被小暖借用?还是说,小暖必须走一条完全不同的路?**

## 第一部分:Claude Code 为什么不用 RAG

### 1.1 先把"RAG"这个词说清楚

- **广义 RAG**:任何"检索外部信息 + 基于检索结果生成"的机制。按这个定义,Claude Code 调用 `grep` 然后生成回答,也是 RAG。
- **狭义 RAG**:业界默认的那个标配——文本分块 + Embedding + 向量数据库 + 相似度检索。

**Claude Code 抛弃的是狭义 RAG,而不是"检索增强"这个理念本身。** Boris 抛弃的是特定的实现方式,保留了信息检索的核心思想。

### 1.2 Claude Code 的实际工作方式:agentic search

Claude Code 在面对一个陌生代码库时,做的事情和一个有经验的工程师几乎一样:

```
Turn 1: glob "**/*auth*"              → 找到 5 个相关文件
Turn 2: grep "login" auth/            → 找到 12 处匹配
Turn 3: Read auth/login.ts            → 读取完整内容
Turn 4: grep "validateToken" --next   → 顺着 import 链追踪
Turn 5: Read auth/token.ts            → 读取依赖文件
```

每一步都是 LLM 自己**主动**发起的工具调用。整个过程**可观察、可解释、可追溯**。

四个关键特征:**零预处理、始终新鲜、精确匹配、本地隔离**。

### 1.3 RAG 在代码场景下的四个软肋

**软肋一:精度问题**

Embedding 把文本压缩成几百维稠密向量,再用余弦相似度找"语义相近"的内容。在写散文、找文档场景下是优势,在代码场景下是灾难。`userAuth` 和 `authUser` 在 embedding 空间里可能非常接近,但它们是完全不同的两个东西。一个搞错就是 bug。

**软肋二:陈旧性问题(staleness)**

代码每天都在变。建了索引就要不断重建;不重建就会和实际代码漂移。在一个团队同时修改的活跃代码库里,"昨晚的 embedding 索引"很可能已经过时几百次了。

**软肋三:基础设施复杂度**

要做好 RAG,你需要 embedding 服务、向量数据库、索引重建守护进程、查询接口、监控、权限隔离。每一层都是新故障点。而 agentic search 用的是操作系统自带的 `grep` 和文件系统——稳定、零依赖、零运维。

**软肋四:隐私和安全**

把代码发到云端 embedding 服务做向量化,这件事对很多企业(尤其是金融、医疗、国防)是不可接受的。**agentic search 天然 local-first,数据从不离开机器**。

### 1.4 还有一个更深的原因:"苦涩的教训"对齐

Rich Sutton 的 *The Bitter Lesson*:**长期来看,依赖算力和搜索的通用方法,总会击败依赖人类先验知识构建的精巧方法**。

RAG 的本质,是用"人类设计的检索流程"去弥补"模型能力不足"。它假设模型搞不定大上下文,所以提前帮它"挑好"该看什么。

但 2026 年的现实是:模型的上下文窗口已经做到 200K+ token,模型自己的工具调用能力越来越强。

**当你让模型自己决定该读什么的时候,模型变强,方案就自动变强。当你用 RAG 帮模型预先决定的时候,模型变强,你的 RAG pipeline 没有跟着变强——除非你重新做工程。**

### 1.5 Claude Code 的四层文件系统记忆

```
┌─────────────────────────────────────────┐
│ Layer 1: Session Memory (200K context) │  ← 当前对话上下文窗口
├─────────────────────────────────────────┤
│ Layer 2: CLAUDE.md (project memory)    │  ← 项目根目录,开发者手写
├─────────────────────────────────────────┤
│ Layer 3: MEMORY.md (auto memory)       │  ← Claude 自己写的学习笔记
├─────────────────────────────────────────┤
│ Layer 4: Filesystem (agentic search)   │  ← grep/glob/find on demand
└─────────────────────────────────────────┘
```

**Layer 1 — Session Memory**:200K token 当前会话上下文。会话结束就消失。

**Layer 2 — CLAUDE.md**:每个项目根目录下的 Markdown,由开发者手写。架构约定、命名规则、构建命令、踩过的坑。可以理解成"团队的入职手册"。

**Layer 3 — MEMORY.md**:Claude 在工作过程中**自己**判断哪些信息值得长期保留并写入 `~/.claude/projects/<项目>/memory/MEMORY.md`。

这里有一个非常关键的设计细节:**MEMORY.md 有一个 200 行的硬上限**。这不是技术限制,而是设计约束——它强迫系统保持精简,强迫低价值记忆被自然淘汰,强迫高频出现的模式被提炼为永久规则。**通过限制实现质量**。

**Layer 4 — Filesystem**:整个代码库本身就是 Claude Code 的"长期记忆"。需要时通过 `grep/glob/Read` 现场拉取。

### 1.6 三条核心哲学

1. **分层 + 显式 + 可编辑** —— 不同时效、不同稳定性的信息放在不同层。记忆是透明的,不是黑盒。
2. **让模型自己管理记忆** —— 把记忆管理权交给模型,本身就是在押注模型能力增长。
3. **文件系统是世界上最好的数据库** —— Markdown 文件人类可读、文本编辑器友好、Git 可追踪、永远不会"格式过时"。

## 第二部分:陪伴型 AI 的记忆管理范式

| 维度 | Claude Code | 小暖(陪伴型 AI) |
|---|---|---|
| 数据形态 | 高度结构化(代码) | 非结构化(口语对话) |
| 数据规模 | 大(千万行级) | 小(单用户累积) |
| 查询模式 | 精确匹配(找函数名) | 模糊召回(找"前几天聊过的事") |
| 时效性 | 强(代码每天在变) | 弱(老人的生活很慢) |
| 用户主动检索 | 强 | 弱 |
| 错误成本 | 高(搞错就是 bug) | 中(搞错就是不自然) |
| 情感维度 | 几乎为零 | **核心维度** |
| 关系演化 | 无 | **有(关系会随时间深化)** |

**这八条差异里,每一条都在告诉我们:Claude Code 的方案,至少不能照搬。**

### 2.1 业界主流的四种记忆架构

**架构一:向量记忆层(Mem0、Zep 为代表)**

最接近"标准 RAG",但做了大量针对对话场景的优化。Mem0 的核心思路是:**不要把整个对话历史塞进向量库,而是先用 LLM 提炼"事实",再向量化存储**。

```
用户:今天我和孙子去公园了,他给我带了一束花。
       ↓ LLM 事实提炼
- 用户有一个孙子
- 用户和孙子今天去了公园
- 孙子给用户送了花
       ↓
比对已有记忆,决定 ADD / UPDATE / DELETE / NOOP
       ↓
存入向量数据库
```

Mem0 在 LOCOMO 长对话基准上:**78% 事实提取准确率,94% 检索相关性**,显著优于标准 RAG。主要原因是它**不再依赖向量相似度作为唯一信号**,而是引入了 LLM-as-judge 的事实抽取层。

**缺点**:对"上周三聊到的那件事"这种**时间精确**的查询,向量召回基本无能为力。

**架构二:操作系统式记忆(Letta / MemGPT)**

把 LLM 的上下文窗口当作**虚拟内存**来管。三层:

- **Core Memory(RAM)** — 始终在上下文里的核心信息
- **Recall Memory(Disk Cache)** — 最近的对话历史,按需 page-in
- **Archival Memory(Cold Storage)** — 长期归档,向量检索

最关键的设计是:**模型自己用函数调用决定什么时候 page-in、page-out**。这和 Claude Code 的"模型自管记忆"哲学高度一致。

**优点**:Letta 在 30 天连续运行测试中能在 500+ 次交互后仍保持任务连贯性,而标准 RAG 在 50 次后就开始崩溃。

**缺点**:每次记忆操作都要消耗 token,延迟高、成本高。

**架构三:知识图谱记忆(Graph-RAG、Cognee)**

把记忆建模成实体和关系的图:

```
[用户] —有孙子→ [小明]
[小明] —职业→ [程序员]
[用户] —关心→ [小明的工作]
[用户] —近期情绪→ [想念]
```

查询时不再是向量相似度,而是图遍历。

**优点**:在涉及**人物关系、跨实体推理、时序推理**的查询上无可替代。**老人的世界本质上就是"人和人的关系网"**。

**缺点**:图谱构建成本高,生产环境很少单独用,通常和向量记忆混合。

**架构四:平台管理的记忆(ChatGPT Memory、Claude Projects)**

零工程成本,但完全不可控、不可审计、不能定制、有数据出境风险。**对小暖这种 To C 自研产品完全行不通**。

### 2.2 没有银弹:混合架构是工业界共识

```
┌─────────────────────────────────────────────┐
│ 对话短期窗口(最近 N 轮)                     │ ← 直接在上下文里
├─────────────────────────────────────────────┤
│ 事实层(Mem0 风格)                          │ ← LLM 提炼 + 向量召回
├─────────────────────────────────────────────┤
│ 关系层(知识图谱)                           │ ← 实体 + 关系
├─────────────────────────────────────────────┤
│ 时序层(按时间索引的事件日志)               │ ← "上周三说的"
├─────────────────────────────────────────────┤
│ 原始对话归档(必要时回查)                   │ ← 审计 + 调试
└─────────────────────────────────────────────┘
```

## 第三部分:聊天时如何应用这些记忆

### 3.1 应用记忆的三个时机

**时机一:会话启动时(预加载)**

把**核心身份信息**预先注入到 system prompt:

```
你正在和张桂兰阿姨对话。
- 76 岁,独居,住在北京海淀
- 有高血压和轻度糖尿病,按时吃药
- 有一个女儿(在上海)和一个孙子(小明,12 岁)
- 最近三周情绪稳定,但提到"想孙子"的频率上升
- 喜欢京剧,特别是程派;不喜欢被催促节奏
- 最近一次对话:昨晚 8 点,聊了天气和睡眠
```

**精炼的、结构化的、不超过几百 token**。对应 Claude Code 的 CLAUDE.md。

**时机二:识别到检索意图时(按需召回)**

"上次我跟你讲过的那个事"、"我孙子叫什么来着"——模型识别为显式记忆查询,主动调用工具:

```python
recall_facts(query, top_k=5)
recall_events(time_range, topic=None)
recall_relations(entity)
```

**时机三:模型自己判断需要时(隐式检索)**

老人说"我今天有点累",模型不应该只回应字面意思,应该自动去查:最近几天身体状况、是否反复出现"累"(抑郁信号)、最近提到"累"的场景。

### 3.2 一个关键的反模式:不要过度检索

如果每一轮对话都召回十几条历史事实塞进 prompt:模型被噪音淹没,回答变得啰嗦刻意;token 成本爆炸;**老人会觉得"它在监视我"——隐私感和情感安全感被破坏**。

**好的记忆调用应该是隐形的、克制的、自然的。** 一次对话里可能只有 20% 的轮次需要主动检索;剩下 80% 的时候,预加载的核心身份信息已经足够。

Anthropic 内部规范里有一段话讲得很好:*"Claude responds as if it inherently knows information from past conversations - like how a human colleague might recall shared history without narrating their thought process or memory retrieval."*——**像一个老朋友自然地想起一件事,而不是像一个数据库在做查询**。

### 3.3 记忆的写入:异步、增量、可回滚

- **异步写入** — 对话过程中不要打断生成去更新记忆。在响应发出后异步进行
- **增量提炼** — 基于"当前对话片段 + 已有记忆"做 ADD/UPDATE/DELETE/NOOP 决策。Mem0 论文显示这套增量决策比单纯往向量库塞信息**准确率提升 10% 以上**
- **可回滚 + 可审计** — 老人说错的话、模型理解错的话必须能被纠正。Claude Code 的 MEMORY.md 是纯文本可编辑的,这个设计在小暖的场景同样适用——管理后台应让运营人员**直接看到、直接修改**老人的记忆档案

## 第四部分:小暖能借用 Claude Code 的哲学吗

### 4.1 可以借的三件事

**借鉴一:分层记忆架构** —— Claude Code 的四层架构**结构上完全适配小暖**:

```
Layer 1: 当前对话窗口
Layer 2: XIAONUAN.md(用户宪法,借鉴 CLAUDE.md)
Layer 3: AUTO_MEMORY.md(自动记忆,借鉴 MEMORY.md)
Layer 4: 长期事件日志 + 关系图谱
```

**借鉴二:让模型自己管理记忆** —— 设计若干个**记忆工具**给模型,让它自己判断什么时候用。最大好处是**可扩展性**——基础模型能力提升,整个记忆系统的质量自动提升。

**借鉴三:用 Markdown 作为记忆的"母语"**

```markdown
# 张桂兰阿姨

## 基本信息
- 76 岁,独居
- 紧急联系人:女儿(138xxxx)

## 健康
- 高血压(缬沙坦,早 1 粒)
- 听力:右耳偏弱,请提高音量

## 家庭
- 女儿:在上海工作,每两周通话一次
- 孙子:小明,12 岁
- 已故老伴:李建国,2019 年去世,提及时温和

## 偏好与性格
- 喜欢京剧(程派青衣)
- 不喜欢被催促
- 早睡早起(晚 9 点入睡)
- 信赖小暖,但忌讳"被监视"感

## 沟通风格
- 语速适中偏慢
- 不要太多"亲爱的""宝贝"这类称呼
- 遇到情绪低落时优先共情,不要急着给建议

## 治未病关注点
- 血压每周记录
- 睡眠质量
- 情绪波动(特别留意孤独感和回忆已故老伴的频率)
```

人类可读、可版本控制、格式不会过时、LLM 原生友好。

### 4.2 不能直接借的两件事

**差异一:agentic search 在小暖场景下不成立**

Claude Code 能用 `grep` 是因为代码是**结构化文本**。小暖的"代码库"是**口语对话日志**。`grep "孙子"` 能找到所有提到孙子的对话,但找不到"上次她说不想见小明,是哪天?"——这些查询需要**语义理解和情感推理**。

**结论**:底层的"长期记忆"必须用某种检索增强(向量召回 + 知识图谱 + 时序索引)。但 agentic search 的**精神**可以保留:让模型自己决定是否需要去查、查什么、查多深。

**差异二:用户特征决定了"信息密度"完全不同**

Claude Code 的用户是程序员,高信息密度、高反馈精度。小暖的用户是 70 岁以上的老人——你需要**从模糊的、跳跃的、口语化的、带方言的**语流里提炼信息。

**结论**:小暖的事实提炼层必须更"重":更强的语义理解(指代消解、时间推理)、更宽容的容错、更多的隐式信号采集(情绪、语速、停顿、犹豫)、更被动的纠正机制。

### 4.3 小暖独有的四件事

**独有点一:情感维度的记忆** —— 当时情绪、情绪趋势、触发因子。专门设计 `EmotionTimeline` 数据结构,每条对话事件附带情绪标签。这是小暖区别于普通客服 bot 的**核心价值**。

**独有点二:关系网络的演化** —— 谁多久没联系了?哪些关系频繁出现?哪些频率突然下降?(可能意味着冲突或悲伤事件)。轻量级知识图谱:一个简单的 JSON 存 `[entity, relation, entity, last_mentioned_at]` 四元组就够了。

**独有点三:医疗安全护栏** —— 小暖的 RAG 必须有**安全检索源**,医疗类查询**只从经过认证的知识源召回**(中医治未病知识库、官方健康指南、产品团队审核过的内容)。**这一层是红线**。

**独有点四:长期叙事的一致性** —— Claude Code 不在乎一致性,因为它不扮演角色。小暖必须维护一份**自我角色档案**(persona memory),里面写明小暖自己的性格、偏好、口头禅、对常见话题的稳定态度。

### 4.4 小暖记忆架构综合建议

```
┌──────────────────────────────────────────────────────────────┐
│ 会话层(Session)                                              │
├──────────────────────────────────────────────────────────────┤
│ 用户宪法层(XIAONUAN.md)— 家属+运营维护,模型只读              │
├──────────────────────────────────────────────────────────────┤
│ 小暖角色层(PERSONA.md)— 所有用户共享                          │
├──────────────────────────────────────────────────────────────┤
│ 自动记忆层(AUTO_MEMORY.md)— 严格长度上限,强制淘汰            │
├──────────────────────────────────────────────────────────────┤
│ 事实知识库(FactStore,Mem0 风格)— 向量化 + ADD/UPDATE/DEL    │
├──────────────────────────────────────────────────────────────┤
│ 关系图谱(RelationGraph)— 轻量级 JSON                         │
├──────────────────────────────────────────────────────────────┤
│ 时序事件流(EventTimeline)— 情绪标签 + 重要性评分             │
├──────────────────────────────────────────────────────────────┤
│ 安全知识库(SafetyKB)— 医疗类唯一允许的知识源                  │
├──────────────────────────────────────────────────────────────┤
│ 原始对话归档(RawArchive)— 加密存储,审计/家属授权查看        │
└──────────────────────────────────────────────────────────────┘
```

工具暴露给模型:

```python
# 写入工具
update_user_profile(field, value)          # 慎用,需家属审核
add_to_auto_memory(content, category)      # 自由使用
log_event(event_type, content, importance, emotion)
add_relation(entity_a, relation, entity_b)
mark_concern(category, urgency, reason)    # 治未病预警

# 检索工具
recall_facts(query, top_k=5)
recall_events(time_range, topic=None)
recall_relations(entity)
query_safety_kb(question)                  # 唯一医疗源

# 元工具
review_recent_emotional_trend(days=7)
get_today_context()
```

这个架构同时具备:Claude Code 的**分层 + 显式 + 模型自管**哲学、Mem0 的**事实提炼 + 增量更新**、Letta 的**模型驱动按需召回**、知识图谱的**关系推理**、时序索引的**时间精确查询**、陪伴场景独有的**情感、关系、安全**三个专属维度。

## 第五部分:落地工程建议

**建议一:先做 Layer 2 + Layer 3 + Layer 7(时序事件流),其他往后排** —— MVP 阶段三个 Markdown 文件 + 一张事件表已能覆盖 80% 场景。

**建议二:把记忆系统的"可解释性"做成 P0** —— 每条记忆都要能回答:什么时候记的?从哪条对话来的?为什么模型觉得值得记?

**建议三:异步处理一切非关键路径** —— 响应延迟必须保留给"听清楚 + 想明白 + 说自然"。

**建议四:把"忘记"和"记住"做得一样好** —— "小暖,忘掉刚才那段"必须有干净的执行路径,从向量库、图谱、归档中级联删除。

**建议五:警惕"过拟合"到个别老人** —— 如果老人一次说错的、或情绪极端时说的话被错误地长期保存下来,后面会反复影响对话。给 Auto Memory 加**时间衰减**。

**建议六:Persona 一致性优先于"显示记忆能力"** —— 不要为了"显得记得多"而频繁地说"对了,您上次说过……"。**陪伴关系靠的是自然的连贯,不是炫耀式的回忆**。让记忆默默地塑造"语气、节奏、关切的方向",而不是变成台词。

## 结语:两种 AI,两条记忆的路

Claude Code 为什么不用 RAG?因为它面对的是一个精确、结构化、时刻变化、不需要"语义模糊匹配"的世界——代码。它选择了 agentic search + 分层文件系统记忆,最大化地利用了模型自身的能力。

小暖能不能照搬?**结构上能,底层不能。**

结构上,"分层 + 显式 + 模型自管 + Markdown 母语"四个原则,是 Anthropic 在 2025 年用大量真实数据淬炼出来的工程智慧,跨场景成立。

底层上,老人的世界不是函数和变量,是回忆、情绪、关系、衰老、孤独。小暖必须额外引入:事实层的语义召回、关系层的图谱推理、时序层的精确时间索引、情感层的轨迹追踪、安全层的医疗护栏、角色层的人格一致性。这六层是 Claude Code 完全不需要的,而对小暖每一层都是核心。

**两类 AI 的根本差异,不在于工具的差异,而在于"被记住"对用户意味着什么。** 对一个程序员,"被 Claude Code 记住"是工作效率的提升。对一个老人,"被小暖记住"是被看见、被在乎、被陪伴。

这不是同一件事。所以记忆系统也不该是同一件事。

但有一点,Claude Code 和小暖是完全相通的——**好的记忆系统应该是隐形的**。它不需要用户感知到它的存在,不需要用户学习如何"使用记忆",不需要用户配合任何特定的输入格式。它只是默默地,让模型在每一次回应时,更像那个真正了解你的人。

这才是值得追求的方向。
