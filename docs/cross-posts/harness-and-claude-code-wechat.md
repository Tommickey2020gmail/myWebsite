## 一个让人沮丧的现实

很多人第一次搭 AI agent，都会陷入同一个执念：**选模型**。GPT 还是 Claude？Opus 还是 Sonnet？

然后现实给一记耳光：同样的模型，别人搭出来稳如老狗，自己搭出来动不动跑偏、死循环、改一个文件碰坏三个。

问题不在模型，在一个很多人没听过的东西——**harness（框架 / 脚手架）**。

## 什么是 harness？

**LLM 本身不是 agent。** 模型只做一件事：给它文字、它接着生成文字。它不会自己读文件、跑命令、调 API。

那"会自己干活"的能力从哪来？**包在模型外面的那层软件——就是 harness。** 它负责把需求组织成 prompt、解析"我要调用某工具"的意图、真的去执行、把结果再喂回去、管理上下文、出错重试。

> **模型是发动机，harness 是整辆车。** 一台 F1 发动机装在没有刹车和方向盘的架子上，还不如一辆设计良好的家用车能跑。

## Claude Code 是什么？

**Claude Code 本身就是一个 harness**——Anthropic 已经造好的、针对编程和智能体任务高度调优的具体实现。

它跑在终端里（也有 VS Code / JetBrains / 桌面 App / iOS）。你用大白话告诉它要什么，它自己读代码、跑 shell、改文件、管 git、调外部服务。

所以正确的对比不是"harness vs Claude Code"，而是：

> **用 Anthropic 造好的现成 harness（Claude Code） vs 自己造 / LangGraph / AutoGen / CrewAI**

## 那个反直觉的核心发现

如果这篇文章只记住一句话：

> **harness 对 agent 实际表现的影响，常常比换模型还大。**

只改 harness、模型权重不动的实验结果：

- **斯坦福 & 清华**：同一模型，不同 harness，复杂任务上 **6 倍**性能差距
- **Pi Research**：仅改编辑工具格式，SWE-bench 从 **6.7% 跃升到 68.3%**
- **Vercel**：删 80% 工具，成功率 80% → 100%，延迟 724s → 141s
- **LangChain**：仅改 harness，Terminal-Bench 从 52.8% → 66.5%
- **Princeton CORE-Bench**：同一模型，scaffold A 得 42%，scaffold B 得 78%
- **SWE-bench Pro**：scaffold 质量可造成 **22 分**摆动——好 harness 的中端模型能打败差 harness 的前沿模型

一篇综述（《Agent Harness for Large Language Model Agents》，梳理 110+ 篇论文）直接断言：**harness，而非模型，才是真实世界 agent 系统性能的约束瓶颈。**

最精炼的总结：

> **模型代表性能天花板，harness 决定你能多接近那个天花板。**

## harness 由什么组成

把它拆开就是这五件事：

1. **工具设计**——少而清晰，胜过多而模糊。Vercel 案例：删 80% 反而更好
2. **循环与规划结构**——结构化"规划 → 执行 → 观察"循环的提升，比换更贵的模型还大
3. **上下文管理**——撑过 4 小时会话还是中途"失忆"的分水岭
4. **失败处理**——撞墙能停下来换路的 agent，远比"一条道走到黑"的强
5. **输出校验**——真 scaffold 会管理上下文、处理失败、校验输出；"wrapper" 只是薄薄包一层

## Claude Code 的五层设计

Claude Code 把能力分成五层，每层"上下文成本"不同：

- **CLAUDE.md**——项目宪法，始终加载，放每一轮都必须为真的规则
- **Skills**——可复用工作流，调用时才加载，放偶尔才需要的流程（部署清单、debug playbook）
- **MCP servers**——按需连接的外部工具和数据
- **Subagents**——隔离上下文的子智能体，处理嘈杂的研究 / 审查工作
- **Hooks**——生命周期事件上的确定性脚本，工具调用前后触发

判断法则：**每轮都成立的规则 → CLAUDE.md；偶尔需要的流程 → skill；必须强制的脚本 → hook。**

**Hooks 的哲学意味值得单独强调：** 它们是确定性脚本，不像依赖模型"理解"的 prompt——**它们不会幻觉**。没有 hooks，你的每一道安全防线都依赖模型听懂你的话；有了 hooks，你在系统层强制执行（比如在 `rm -rf` 真正执行前就拦截）。

> **该交给模型判断的交给模型，该用代码兜底的用代码兜底。** 这是搭生产级 agent 的核心心法。

## 两条路怎么选

- **写代码、调系统、做开发**——Claude Code 几乎是当前最强的现成 harness，直接用
- **做面向最终用户的业务 agent**（语音助手 / 客服 / 领域 workflow）——Claude Code 是给开发者在终端用的，不是给终端用户跑业务用的。这时你需要 LangGraph / Dify 这类通用编排

**关键是这两者不冲突，是不同的层。** 你完全可以用 Claude Code 来**开发**那个最终跑在 LangGraph 上的业务 agent。**开发期的 harness 和运行期的 harness，可以是两个东西。**

## 一个重要的反面

数据太亮眼容易让人走另一个极端："模型不重要了"。这是错的。

**规律是：harness 的边际收益，在"任务长、多步、易失败"的场景最大；在短任务、定义清晰的场景里，模型能力本身就够用，harness 的差异被淹没。**

还有一个对所有人都该警醒的 METR 数据：**通过基准测试的 PR，比真实可被合并进主干的 PR，人工合并率低了 24.2 个百分点，而且差距还在以每年 9.6 个百分点的速度扩大。**

> **benchmark 跑得漂亮，不等于真能落地。**

**在你自己的真实任务上测，而不是看排行榜。** 赢了你 benchmark 的那个配置，未必赢得了你的周二上午。

## 该学什么

**学 harness 的本质**（可迁移到任何工具）：
- 上下文工程，而非 prompt 工程
- 工具设计的"少即是多"
- 失败处理与控制流
- 确定性 vs 概率性的边界
- 评测的有效性

**学 Claude Code 的杠杆**（顺序建议）：
1. CLAUDE.md + 权限 → 立项目规则和安全边界
2. MCP → 接内部系统、数据库、私有 API
3. subagents + hooks → 把审查 / 测试 / 安全做成专门 agent 和确定性钩子
4. Plugins → 团队复用，变成团队标准

**铁律：需要永远执行的事，用 hooks，不用 prompt。**

## 一个比读十篇文章更有用的练习

> **选同一个真实任务，分别用 Claude Code 的 subagents、用 LangGraph、用 CrewAI 各跑一遍。记录四个维度：token 消耗、轮次、可靠性、可调试性。**

一次这样的实验，胜过读十篇对比文章。

## 结语

AI agent 这两年的故事，正在从"模型竞赛"悄悄转向"系统竞赛"。

对初学者：**别再只盯着模型的旋钮，认识到 harness 这个更大的旋钮的存在，你就已经领先大多数人了。**

对实践者：**Claude Code 给了你一辆造工精良的车，但真正的功力，在于你懂得它的方向盘、刹车和变速箱各自该在什么时候用——以及，知道什么时候这辆车并不适合你要走的路。**

> **模型决定你能走多远，harness 决定你能不能走到。**
