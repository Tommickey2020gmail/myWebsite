## 一句话开场

**2024 是"RAG 之年",2025 是"流派分化年",2026 RAG 正在被泛化为"上下文工程(Context Engine)"。** 它没有被长上下文取代,反而成为 Agent 时代的核心基础设施。

## 从 Lewis 论文到 Context Engine:六年六阶段

| 阶段 | 时间 | 关键事件 |
| --- | --- | --- |
| 学术前史 | 2017-2019 | DrQA、ORQA、BERT-QA |
| RAG 提出 | 2020 | REALM、Lewis RAG、DPR |
| 工程萌芽 | 2023 | LangChain / LlamaIndex 标准化 |
| RAG 之年 | 2024 | GraphRAG、Contextual Retrieval、Self-RAG、Ragas |
| 流派分化 | 2025 | Agentic RAG、LazyGraphRAG、Adaptive RAG |
| 上下文工程 | 2026— | Context Engine、MCP 工具检索 |

2022 年 ChatGPT 暴露大模型的"四宗罪":**知识截止、私域无知、幻觉严重、不可追溯**。RAG 成了"最直接的补丁"——无需 fine-tuning,把企业知识切片入库 + 检索拼接到 prompt,就能让模型回答原本不知道的事。

## RAG 的底层骨架:一句话

> 离线索引(解析 → 清洗 → 切片 → 嵌入 → 入库) → 在线检索(查询改写 → 混合检索 → 元数据过滤 → 重排) → 生成(提示构造 → 引用强制 → 后置校验 → 评估闭环)

三个**最高 ROI 的优化点**:

1. **文档解析(Parsing)** —— "第一道生死关",解析错了后面全白做。投资在解析的回报往往**高于换更贵的嵌入模型**。
2. **Hybrid Search(混合检索)** —— 稠密向量 + 稀疏 BM25 用 RRF 融合,**通常比纯向量准确率高 15-30%**。对专有名词、产品型号、错误码尤其关键。
3. **Reranker(重排)** —— Cross-Encoder 或 ColBERT 做精排,被业内公认**性价比最高的 RAG 优化**。

## 六大流派一张表

| 流派 | 复杂度 | 单次成本 | 延迟 | 适合 | 不适合 |
| --- | --- | --- | --- | --- | --- |
| Naïve RAG | ★ | $0.001 | <1s | FAQ、PoC | 全局/多跳/精确名词 |
| Advanced RAG | ★★ | $0.005 | 1-2s | 企业知识问答 | 复杂跨文档推理 |
| Modular RAG | ★★★ | $0.005-0.02 | 1-3s | 平台层、多业务 | — |
| GraphRAG | ★★★★ | 构建昂贵 | 2-5s | 关系/全局/多跳 | 实时性强、简单事实 |
| Agentic RAG | ★★★★ | $0.02-0.10 | 5-30s | 复杂推理、研报 | 简单事实、成本敏感 |
| Adaptive | ★★★★★ | 按路由 | 按路由 | 全场景平台 | 小团队、初期 |

**Naïve RAG 在生产环境失败率高达 40%**。能跑的 PoC 离生产可用有 10 倍距离。

**别迷信单一架构** —— 一个成熟的企业 RAG 系统几乎都是混合架构。Adaptive RAG(自适应路由)是 2025-2026 的"集大成"模式:简单事实走 Advanced,关系查询走 GraphRAG,复杂多跳走 Agentic,结构化数据走 Text-to-SQL,无需检索直接走 LLM 参数化记忆。

## 长上下文会不会取代 RAG?

不会。Self-Route 类研究的结论很明确:**简单问题用 RAG,复杂综合任务用长上下文,由模型自己决定**。在 TB 级知识库、多用户并发、强权限隔离场景下,RAG 仍是经济上不可替代的方案。

长上下文反而让 RAG 更好用——chunking 可以更粗、生成阶段可以塞更多候选,显著降低召回噪声敏感度。

## 视觉文档 RAG 正在取代 OCR+文本 RAG

2024-2025 最明显的趋势之一:**ColPali / ColQwen-VL 这类"视觉文档"模型直接用 VLM 处理整页 PDF/PPT 截图,跳过 OCR 环节**。尤其在表格、公式、图表密集的文档上,效果显著优于"OCR + 文本 RAG"。

## 四类场景的差异化选型

| 维度 | 个人助理 | 老人陪伴 | 项目管理 | 企业客服 |
| --- | --- | --- | --- | --- |
| 核心流派 | Advanced + LTM | Advanced + Guardrail + CRAG | **GraphRAG** + Advanced + Agentic | Modular + Advanced + Adaptive |
| 是否需图谱 | 否 | 否 | 强需要 | 弱 |
| 是否 Agent | 轻 | 极轻(避免) | 中 | 选择性 |
| 延迟 | <2s | <3s(语音) | <5s | <2s(高并发) |
| 首要风险 | 隐私 | 医疗安全 | 权限泄露 | 幻觉/合规承诺 |
| 关键评测 | 满意度 | 安全率/拒答得当率 | 引用/关系准确率 | Deflection/AHT/CSAT |

**几条关键经验:**

- **个人助理**:静态资料层 + 动态记忆层独立索引。避免过度 Agent 化——90%+ 查询是简单事实。
- **老人陪伴**:**纯闲聊路径绕开 RAG**;医疗查询启用"高置信度 + 强制权威 + 拒答兜底"——宁可不答也不能给错误医疗建议。
- **项目管理**:必须用知识图谱。"按状态/负责人筛选"应路由到 Text-to-SQL,不要硬走向量检索。会议纪要尤其重要——很多决策只存在于会议中。
- **企业客服**:Hybrid Search + 强 Rerank 是底线。**宁可"我没找到相关信息,请稍候为您转人工"也不要编造**。

## 七个最常踩的坑

1. 迷信 Naïve PoC
2. Chunk 策略一刀切(合同、手册、对话、代码应有不同策略)
3. **忽视 Parsing(隐形杀手)**
4. 没有评测就调优
5. **过度 Agent 化**(成本/延迟是 Advanced 的 5-20 倍)
6. 把权限和合规留到上线后再补(Day 1 就要做架构设计)
7. 把 RAG 当作"上线就完事"

## RAG 的真正本质

RAG 从来不是单一算法,而是一种"**让大模型与真实世界知识动态对齐**"的工程哲学。

流派的选择不是技术信仰之争,而是基于业务场景、查询模式、成本预算、合规要求的工程权衡。任何架构建议都不是"最优解",而是在多约束下"足够好且可演进"的起点。

**真正的 RAG 工程,始于场景理解,成于持续迭代。**

## 2026 及以后的五条趋势

1. **上下文工程将取代 RAG 成为更准确的描述词**——把知识、工具、记忆、画像检索统一为 Agent 上下文供给体系
2. **视觉文档 RAG 取代 OCR+文本 RAG**
3. **GraphRAG 与 Agentic RAG 边界模糊**——Agent on Graph 成为常见模式
4. **评测和可观测性变得和模型本身同等重要**
5. **端侧/小模型 RAG 因隐私和成本驱动迎来新发展**——老人陪伴、个人助理类场景尤为典型
