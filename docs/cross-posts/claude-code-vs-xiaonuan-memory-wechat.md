## 一个被忽略的反常识

2026 年 1 月,Claude Code 团队负责人 Boris Cherny 在 Hacker News 上说:

> "Claude Code 早期版本用过 RAG + 本地向量数据库,但我们很快发现 **agentic search 在大多数情况下都更好**。"

在一个"言必称 RAG"的时代,这句话像是从行业主流路线上划下了一道斜线。Claude Code 选择了一条更"原始"的路:**不建索引、不做向量化,靠 `grep`、`glob`、`find`、`Read` 这些 Unix 老兵命令行工具,像真正的程序员那样一行一行探索代码库**。

这件事真正值得深思的不是"RAG 是否过时",而是它逼着我们重新审视 **"AI 记忆"这件事究竟该怎么做**。

我们正在构建的小暖,是一个面对老人的语音陪伴系统——它要记住的不是函数签名,而是张阿姨上周提过的孙子叫什么、李大爷昨天血压高了三天没说话、王奶奶最近开始反复讲同一个故事。

**Claude Code 的方案能不能借给小暖?这篇文章试着把这个问题讲清楚。**

## Part 1:Claude Code 为什么不用 RAG

先消除歧义:Claude Code 抛弃的是**狭义 RAG**(分块 + Embedding + 向量库 + 相似度检索),不是"检索增强"这个理念本身。

它的实际工作方式叫 **agentic search**:

```
glob "**/*auth*" → 找到 5 个文件
grep "login" → 找到 12 处匹配
Read auth/login.ts → 读取完整内容
grep "validateToken" → 顺着 import 链追踪
```

每一步都是 LLM 自己主动发起的工具调用——**零预处理、始终新鲜、精确匹配、本地隔离**。

### RAG 在代码场景下的四个软肋

1. **精度问题**:`userAuth` 和 `authUser` 在 embedding 空间里很近,但完全是两个东西
2. **陈旧性**:代码每天在变,索引和实际代码必然漂移
3. **基础设施复杂度**:embedding 服务、向量库、索引重建守护进程、查询接口、监控、权限隔离——每一层都是新故障点
4. **隐私和安全**:代码发到云端做向量化,对很多企业不可接受

### 一个更深的原因:对齐"苦涩的教训"

Rich Sutton 的 *The Bitter Lesson*:**长期来看,依赖算力和搜索的通用方法,总会击败依赖人类先验知识构建的精巧方法**。

RAG 的本质是用"人类设计的检索流程"去弥补"模型能力不足"。但 2026 年的现实是模型上下文已经 200K+ token、工具调用能力越来越强。

**当你让模型自己决定该读什么的时候,模型变强,方案就自动变强。当你用 RAG 帮模型预先决定的时候,模型变强,你的 RAG pipeline 没有跟着变强——除非重新做工程。**

### Claude Code 的四层文件系统记忆

```
Layer 1: Session Memory(200K 上下文)
Layer 2: CLAUDE.md(项目宪法,开发者手写)
Layer 3: MEMORY.md(自动记忆,Claude 自己写,200 行硬上限)
Layer 4: Filesystem(agentic search,grep/glob on demand)
```

关键的设计细节:**MEMORY.md 有 200 行的硬上限**。不是技术限制,而是设计约束——**通过限制实现质量**,强迫低价值记忆被自然淘汰。

三条核心哲学:

- **分层 + 显式 + 可编辑** — 记忆透明,不是黑盒
- **让模型自己管理记忆** — 押注模型能力增长
- **文件系统是世界上最好的数据库** — Markdown 人类可读、Git 可追踪、永远不会"格式过时"

## Part 2:小暖的场景为什么完全不同

| 维度 | Claude Code | 小暖 |
|---|---|---|
| 数据 | 结构化代码 | 口语对话 |
| 查询 | 精确匹配 | 模糊召回 |
| 时效 | 强(代码每天变) | 弱(老人生活很慢) |
| 错误成本 | 高(bug) | 中(不自然) |
| **情感维度** | 几乎为零 | **核心维度** |
| **关系演化** | 无 | **随时间深化** |

业界主流的四种陪伴记忆架构:

- **Mem0 / Zep(向量记忆)** — 先 LLM 提炼事实再向量化。LOCOMO 基准 78% 事实提取准确率,显著优于标准 RAG。缺点:对"上周三那件事"这种时间精确查询无能
- **Letta / MemGPT(操作系统式)** — 把上下文窗口当虚拟内存,模型自己 page-in/out。30 天 500+ 交互仍保持连贯,标准 RAG 50 次就崩溃。缺点:成本高
- **GraphRAG(知识图谱)** — 对老人陪伴尤其重要,**老人的世界本质上就是"人和人的关系网"**
- **平台管理记忆(ChatGPT/Claude Projects)** — 零工程成本,但不可控、不可审计、有数据出境风险。**To C 自研产品完全行不通**

**2026 年业界共识:没有银弹,生产环境几乎都是混合架构。**

## Part 3:应用记忆时的一个反模式

陪伴 AI 在一次对话中调用记忆有三个时机:**预加载(会话启动) + 按需召回(识别检索意图) + 隐式检索(模型自判)**。

但是,**最常见的工程错误是过度检索**:

如果每一轮对话都召回十几条历史事实塞进 prompt:
- 模型被噪音淹没,回答变得啰嗦、刻意
- token 成本爆炸
- **老人会觉得"它在监视我"——情感安全感被破坏**

**好的记忆调用应该是隐形的、克制的、自然的。** 一次对话里可能只有 20% 的轮次需要主动检索;剩下 80% 的时候,预加载的核心身份信息已经足够。

Anthropic 内部记忆规范里有一段话讲得很好——**像一个老朋友自然地想起一件事,而不是像一个数据库在做查询**。

## Part 4:小暖能借什么、不能借什么

### 能借的三件事

**借鉴一:分层 Markdown 记忆架构**

```
Layer 1: 当前对话窗口
Layer 2: XIAONUAN.md(用户宪法,家属+运营维护,模型只读)
Layer 3: AUTO_MEMORY.md(自动记忆,严格长度上限)
Layer 4: 长期事件日志 + 关系图谱
```

XIAONUAN.md 大概长这样:

```markdown
# 张桂兰阿姨
## 基本信息
- 76 岁,独居,紧急联系人:女儿
## 健康
- 高血压(缬沙坦,早 1 粒)
- 听力:右耳偏弱,请提高音量
## 家庭
- 已故老伴:李建国,2019 年去世,提及时温和
## 沟通风格
- 不要太多"亲爱的"称呼
- 遇到情绪低落时优先共情,不要急着给建议
## 治未病关注点
- 血压每周记录;情绪波动(特别留意孤独感)
```

**借鉴二:把记忆管理权交给模型** — 设计 `add_to_auto_memory`、`log_event`、`add_relation` 等工具,让模型自己判断什么时候调用。最大好处是**可扩展**——基础模型变强,记忆质量自动变强。

**借鉴三:用 Markdown 作为记忆的"母语"** — 人类可读、可版本控制、格式不会过时、LLM 原生友好。

### 不能直接借的两件事

**差异一:agentic search 在小暖不成立** — `grep "孙子"` 能找到所有提到孙子的对话,但找不到"上次她说不想见小明,是哪天?"——需要**语义理解和情感推理**。底层必须用向量召回 + 知识图谱 + 时序索引。但 agentic search 的**精神**可以保留:让模型自己决定是否查、查什么、查多深。

**差异二:用户信息密度完全不同** — Claude Code 的用户是程序员,**高信息密度、高反馈精度**。小暖的用户是老人,你需要从**模糊的、跳跃的、口语化的、带方言的**语流里提炼信息。事实提炼层必须更"重":更强的指代消解和时间推理、更宽容的容错、更多隐式信号采集(情绪、语速、停顿、犹豫)。

## Part 5:小暖独有的四件事

Claude Code 完全不需要,但小暖**必须**:

1. **情感维度的记忆** — `EmotionTimeline`,每条事件附带情绪标签,按时间维度可视化。这是小暖区别于客服 bot 的**核心价值**
2. **关系网络的演化追踪** — 谁多久没联系了?哪些关系频率突然下降?(可能是冲突或悲伤事件)。轻量级 JSON 图谱即可
3. **医疗安全护栏** — 医疗类查询**只从经过认证的知识源召回**(治未病知识库、官方健康指南)。绝不能让模型用通用知识"自由发挥"医疗建议。**这是红线**
4. **长期叙事一致性** — 小暖需要维护一份 PERSONA.md,所有用户共享,写明自己的性格、口头禅、稳定立场

## Part 6:小暖的九层混合架构

```
1. 会话层(Session)
2. 用户宪法层(XIAONUAN.md)
3. 小暖角色层(PERSONA.md)
4. 自动记忆层(AUTO_MEMORY.md)
5. 事实知识库(Mem0 风格)
6. 关系图谱(轻量 JSON)
7. 时序事件流(EmotionTimeline)
8. 安全知识库(医疗类唯一来源)
9. 原始对话归档(加密 + 家属授权查看)
```

同时具备:Claude Code 的**分层 + 显式 + 模型自管**、Mem0 的**事实提炼 + 增量更新**、Letta 的**模型驱动按需召回**、知识图谱的**关系推理**、时序索引的**时间精确查询**、陪伴场景独有的**情感、关系、安全**三个专属维度。

## 几条工程落地建议

- **先做 Layer 2 + 3 + 7,其他往后排** — MVP 三个 Markdown + 一张事件表能覆盖 80% 场景
- **可解释性做成 P0** — 每条记忆要能回答:什么时候记的?从哪条对话来的?为什么觉得值得记?
- **异步处理一切非关键路径** — 响应延迟必须保留给"听清楚 + 想明白 + 说自然"
- **把"忘记"做得和"记住"一样好** — "小暖,忘掉刚才那段"必须从向量库、图谱、归档级联删除
- **警惕过拟合到个别老人** — 加时间衰减,长期未复现的"事实"自然淡化
- **Persona 一致性优先于"显示记忆能力"** — 不要为了显得记得多而频繁说"对了,您上次说过……"。**陪伴关系靠的是自然的连贯,不是炫耀式的回忆**

## 结语

Claude Code 为什么不用 RAG?因为它面对的是精确、结构化、时刻变化、不需要"语义模糊匹配"的世界——代码。

小暖能不能照搬?**结构上能,底层不能。**

底层上,老人的世界不是函数和变量,是回忆、情绪、关系、衰老、孤独。

**两类 AI 的根本差异,不在于工具的差异,而在于"被记住"对用户意味着什么。** 对一个程序员,"被 Claude Code 记住"是工作效率的提升。对一个老人,"被小暖记住"是被看见、被在乎、被陪伴。

这不是同一件事。

但有一点是相通的——**好的记忆系统应该是隐形的**。它不需要用户感知到它的存在,不需要用户学习如何"使用记忆"。它只是默默地,让模型在每一次回应时,更像那个真正了解你的人。
